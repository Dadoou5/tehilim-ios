> Document consolidé. Contributions par agent.

## 1. Chef de projet — Cadrage

### Vision V1.2
Un rappel quotidien optionnel, paramétrable, qui amène l'utilisateur directement à la liste des Tehilim du jour en un tap. Aucune obligation, aucune publicité, aucune télémétrie.

### Périmètre
- ✅ Toggle dans Réglages → "Rappel quotidien".
- ✅ Heure paramétrable via `DatePicker` (au format hh:mm).
- ✅ Permission iOS demandée uniquement au moment du toggle ON.
- ✅ Tap sur la notification → onglet **Aujourd'hui** sélectionné, pile remise à la racine.
- ✅ Bannière explicative + bouton "Ouvrir Réglages iOS" si l'utilisateur a refusé la permission au niveau OS.
- ✅ Reprogrammation automatique à chaque changement d'heure.
- ✅ Re-vérification de la permission au passage en avant-plan (cas où l'utilisateur l'a coupée via Réglages iOS sans rouvrir l'app).

### Hors périmètre V1.2
- Notifications dynamiques avec liste réelle des psaumes du jour dans le corps (nécessiterait une `Notification Service Extension`).
- Notifications "smart" (skip si l'utilisateur a déjà lu).
- Plusieurs créneaux par jour.
- Apple Watch companion.

### Critères d'acceptation
- [x] Toggle persiste après relance.
- [x] Heure persiste après relance.
- [x] Première activation → fenêtre système iOS de demande de permission.
- [x] Refus → bannière + lien vers Réglages iOS.
- [x] Tap sur notif (cold launch et warm) → onglet Aujourd'hui.
- [x] Désactivation → notification annulée, plus de rappel envoyé.
- [x] Build vert + 32/32 tests.

### Risques
- **Permission refusée définitivement** : l'utilisateur doit aller dans Réglages iOS — friction. Mitigé par le bouton direct.
- **Heure d'été / changement de fuseau** : `UNCalendarNotificationTrigger` re-calibre selon le fuseau actuel, c'est natif.
- **App désinstallée puis réinstallée** : la permission est remise à zéro côté iOS, l'utilisateur devra réautoriser.

## 2. Expert Tehilim — Wording

### Choix retenus
- **Titre** : `Tehilim du jour`
- **Corps** : `C'est le moment de lire tes Tehilim du jour.`
- Tutoiement, neutre, pas d'injonction religieuse.
- Heure par défaut : `08:00` — non prescriptive, simple compromis matinal.

### Ce qu'on évite
- "Tu n'as pas encore lu…" (culpabilisation).
- "Pour ta protection, lis…" (promesse).
- Référence à un horaire de prière spécifique (Chaharit / Min'ha / Maariv) — l'utilisateur ne nous a pas donné cette information et les sensibilités varient.

### Validation requise
- [ ] Validation par un porteur référent du wording avant publication.

## 3. UX — Parcours

### Parcours d'activation

```
[Réglages]
   │ tap "Me notifier chaque jour"
   ▼
[iOS demande la permission] (1ère fois uniquement)
   ├─ Autoriser → DatePicker apparaît, programmation immédiate
   └─ Refuser → Toggle revient sur OFF
                                 (l'utilisateur peut retoggler plus tard,
                                  mais iOS ne montrera plus la fenêtre)
```

### Parcours de tap sur notification

```
[Notification reçue à 08:00]
   │ tap
   ▼
[Cold launch ou warm launch de l'app]
   │ NotificationManager → pendingRoute = .daily
   ▼
[RootTabView observe → bascule onglet Aujourd'hui]
   │ pile dailyPath réinitialisée à racine
   ▼
[DailyView → liste des Tehilim du jour selon le mode actif]
```

### États conditionnels
- **Toggle OFF** : section ne montre que le toggle, footer caché.
- **Toggle ON + permission accordée** : DatePicker visible + footer informatif.
- **Toggle ON + permission refusée** : DatePicker caché, bannière d'erreur + bouton vers Réglages iOS.
- **Toggle ON + permission `notDetermined` (jamais demandée)** : ne se produit pas (le toggle ON déclenche la demande).

### Discoverability
- Section "Rappel quotidien" en milieu de Form, après "Lecture quotidienne" — proximité sémantique.
- Pas de promotion forcée à l'onboarding (V1.2 reste discret).

## 5. Designer — Direction visuelle

### Notification système
- Pas de pièce jointe média, pas de bouton d'action custom (V2 si demandé).
- Son par défaut système (.default) — discret.
- Pas de badge sur l'icône (notification informative, pas de "tâche" à cocher).

### Réglages
- Bannière d'erreur en `errorToken` (#A33A2C clair / #E07A6E sombre) — visible mais sans agressivité.
- Bouton bordered (pas filled) → action volontaire mais secondaire à la main UI.
- Footer informatif en `.caption` gris secondaire — explicite la fréquence et la finalité.

## 6. Dev iOS / SwiftUI — Architecture technique

### Fichiers ajoutés
- [`Core/Services/NotificationManager.swift`](../ios/Tehilim/Core/Services/NotificationManager.swift) — singleton, ObservableObject, delegate.

### Fichiers modifiés
- [`Core/Persistence/Preferences.swift`](../ios/Tehilim/Core/Persistence/Preferences.swift) — 3 nouvelles clés (`enabled`, `hour`, `minute`).
- [`Features/Settings/SettingsView.swift`](../ios/Tehilim/Features/Settings/SettingsView.swift) — `NotificationsSection` + `DeniedBanner`.
- [`App/TehilimApp.swift`](../ios/Tehilim/App/TehilimApp.swift) — init du manager au plus tôt.
- [`App/RootTabView.swift`](../ios/Tehilim/App/RootTabView.swift) — observation de `pendingRoute`, hook `scenePhase`.

### API du manager

```swift
@MainActor
final class NotificationManager: NSObject, ObservableObject {
    static let shared: NotificationManager
    @Published private(set) var permission: UNAuthorizationStatus
    @Published var pendingRoute: TabRouter.Tab?

    func refreshPermission() async
    func requestPermission() async -> Bool
    func scheduleDailyReminder(hour: Int, minute: Int) async
    func cancelDailyReminder() async
    func openSystemSettings()
}
```

### Choix techniques
- **Trigger** : `UNCalendarNotificationTrigger(dateMatching: ..., repeats: true)`. iOS gère la récurrence quotidienne sans qu'on ait besoin de re-planifier nous-mêmes.
- **Identifier unique** : `tehilim.daily.reminder`. Un seul rappel actif à la fois.
- **Reprogrammation** : `cancel` puis `add` à chaque changement de toggle ou d'heure.
- **Deep link** : userInfo `["route": "daily"]` dans le content. Le delegate met à jour `pendingRoute`. RootTabView observe et applique.
- **Cold launch** : `NotificationManager.shared` est forcé dans `TehilimApp.init()` pour que le delegate soit en place avant que iOS livre la notification.
- **Foreground refresh** : `onChange(of: scenePhase)` re-calcule `permission` (au cas où l'utilisateur a changé via Réglages iOS).

### Pas de framework tiers, pas d'entitlement supplémentaire
Standard `UserNotifications` framework. Pas besoin de capability spéciale dans Xcode.

## 7. Testeur — Plan de test

### Tests automatisés
Pas de XCTest unitaire pour ce module : la planification dépend du système et n'est pas trivialement testable en CI. Les fonctions internes (formatage heure, mapping de routes) sont triviales et couvertes par compilation.

### Tests manuels obligatoires sur device

| TC | Scénario | Attendu |
|----|----------|---------|
| TC-200 | Toggle ON première fois | iOS affiche le dialogue de permission |
| TC-201 | Permission accordée | DatePicker s'affiche, footer apparaît |
| TC-202 | Permission refusée à la 1ère fois | Toggle revient sur OFF |
| TC-203 | Toggle ON après refus précédent | Bannière "Notifications désactivées" + bouton |
| TC-204 | Tap "Ouvrir Réglages iOS" | App système Réglages s'ouvre sur la fiche Tehilim |
| TC-205 | Activer dans Réglages iOS, revenir à l'app | Bannière disparaît, DatePicker apparaît, programmation auto |
| TC-206 | Modifier l'heure | Notification reprogrammée (vérifiable via `Console` Mac : log de `add` request) |
| TC-207 | Notification reçue (warm) | Bannière in-app + son ; tap → onglet Aujourd'hui |
| TC-208 | Notification reçue (app fermée) | Tap → app lance sur onglet Aujourd'hui |
| TC-209 | Désactiver le toggle | Plus de notification livrée |
| TC-210 | Force quit + relance | Toggle et heure conservés ; notif planifiée toujours active |
| TC-211 | Changement d'heure d'été | Notification respecte l'heure locale (test à H + 6 mois si possible, sinon log) |

### Tests d'edge cases
- Toggle ON + permission jamais accordée + force-quit avant que le dialogue soit traité → état cohérent au relancement.
- Heure modifiée alors que la permission est refusée → reprogrammation tentée silencieusement, pas d'erreur visible.
- App ouverte sur l'onglet Aujourd'hui et notification reçue → pas de boucle (pendingRoute mis à .daily, mais déjà sur Aujourd'hui = no-op visuel).

### Risques résiduels
- Sur simulateur, les notifications fonctionnent mais le tap-to-launch peut être moins fiable que sur device. Tester sur iPhone réel obligatoire avant publication.
- Permission "provisional" (notifications silencieuses) : le code l'accepte comme "autorisée", à confirmer sur device.
